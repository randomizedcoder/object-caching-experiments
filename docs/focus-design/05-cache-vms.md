## 13. Cache VMs: nginx primary and Zot oracle

Both cache VMs run an **identical** config — they are intentionally interchangeable so consistent hashing is meaningful. Everything is installed and configured by NixOS. The cache nginx is **OpenResty** too — not only for uniformity with the clients, but for **headroom**: it leaves the door open to arbitrary Lua logic and, in particular, to a future **upstream / hierarchical cache tier** (parent-child, [§23](08-operations.md#23-future-work)). The moment a cache VM gains an upstream pool of its own, it health-checks and fails over with the **same in-process checker** the clients use ([§11.3](04-client.md#113-health-checking-passive-and-in-process-active)) — no new technology. It runs no checker *today* simply because it has no upstream pool yet; it proxies straight to each origin.

The client-facing vhosts ([§13.2](#132-nginx-wildcard-oci-catch-all)–13.3, [§15](06-mitm-and-content.md#15-llm-model-store-caching)) are **TLS listeners** presenting the shared cache server cert signed by the cache CA ([§11.5](04-client.md#115-encrypted-client-to-cache-hop-tls)); the same cert+key is deployed to both VMs. **nginx is the cache that serves clients**; **Zot ([§13.1](#131-zot-verification-oracle)) runs alongside purely as the verification oracle** ([§7](02-caching-design.md#7-oci-caching-nginx-primary-zot-oracle)) and is never in the client→cache serving path.

### 13.1 Zot (verification oracle)

Kept as the spec-correct ground truth we diff nginx against ([§7.4](02-caching-design.md#74-the-committed-design-nginx-cache-zot-as-verification-oracle)) — *not* a client-facing backend. One Zot instance per Tier-1 upstream (`onDemand` pull-through), with **Prometheus metrics enabled**:

```json
{
  "storage": { "rootDirectory": "/var/lib/cache/zot/docker", "gc": true },
  "http":    { "address": "0.0.0.0", "port": "5050" },
  "log":     { "level": "info" },
  "extensions": {
    "sync": {
      "enable": true,
      "registries": [{
        "urls": ["https://registry-1.docker.io"],
        "onDemand": true,
        "tlsVerify": true,
        "content": [{ "prefix": "**" }]
      }]
    },
    "metrics": {
      "enable": true,
      "prometheus": { "path": "/metrics" }
    }
  }
}
```

- `onDemand: true` = pull-through: on a manifest/blob request, fetch from upstream if absent, store, serve.
- Health: `GET /v2/` answers `200`/`401` — the differential-test harness ([§20](08-operations.md#20-build-and-run-workflow)) pulls against it out of band; the client nginx's in-process Lua health-check ([§11.3](04-client.md#113-health-checking-passive-and-in-process-active)) probes the **nginx** cache (`:8085`), not Zot.
- **Metrics**: `extensions.metrics.enable = true` exposes Prometheus counters (`zot_*`) at `/metrics` on each Zot port — repo pulls, upload counts, sync activity, storage usage. The oracle uses these to compare its hit/miss + storage against nginx's.
- `gcr.io`/`ghcr.io`/`quay.io`/`registry.k8s.io` instances are the same config with their own `rootDirectory`, port (5051–5054), and upstream URL — generated by `nix/modules/zot-oracle.nix` from `constants.upstreams`.

### 13.1.1 What Zot's sync rewrites (and why it matters)

The single most important thing to understand about Zot's pull-through (`sync`) is that, **by default, it does not pass images through byte-for-byte — it converts them.** Most public registries (Docker Hub above all) still serve **Docker schema2** manifests. When Zot syncs such an image it transcodes it to the **OCI** format, and because a manifest's digest is the hash of its own bytes, *transcoding changes the digest*. Verified against the source in `~/Downloads/zot`:

- **Manifest digest is recomputed.** `convertDockerManifestToOCI` builds an OCI manifest and sets its digest to the hash of the *new* bytes: `manifestDesc.Digest = godigest.FromBytes(ociManifestBuf)` (`pkg/extensions/sync/remote.go:384`). The new bytes differ because the manifest body now carries OCI media-type strings — its own `mediaType`, the config descriptor's `mediaType`, and every layer descriptor's `mediaType` are rewritten (`remote.go:362-363`, `:373`) — even though every digest it references is unchanged. That media-type rewrite alone is enough to change the manifest's hash.
- **Config blob is *not* changed — but its descriptor's media type is.** The config digest is re-hashed over the *original* config bytes (`ociManifest.Config.Digest = godigest.FromBytes(configBuf)`, `remote.go:361`), so in practice the **config blob digest is identical** to the origin; only the descriptor's media type flips `application/vnd.docker.container.image.v1+json` → `application/vnd.oci.image.config.v1+json` (`remote.go:362`). (Empirically confirmed: for `alpine:3.9` the config digest matched the origin byte-for-byte.) The manifest-digest change above comes from the media-type rewriting in the manifest body, not from a config-digest change.
- **Index (manifest-list) digest is recomputed.** For multi-arch images each sub-manifest is converted (so each sub-manifest *descriptor's* digest changes per the bullet above) and the index is re-marshalled and re-hashed: `indexDesc.Digest = godigest.FromBytes(indexBuf)` (`remote.go:316`). So the *top-level* digest you'd pin (`image@sha256:…`) changes too.
- **Layer media types are remapped, layer bytes are not.** `toOCIDescriptor` rewrites Docker layer/config/foreign media types to their OCI equivalents (`remote.go:404-419`, e.g. `application/vnd.docker.image.rootfs.diff.tar.gzip` → `application/vnd.oci.image.layer.v1.tar+gzip` at `:415`). The layer **blob digests are preserved** (the compressed tar bytes are untouched), but they sit under a manifest with a different media-type label and a different manifest digest.
- **Foreign / non-distributable layers are skipped.** Layers carrying `urls` (e.g. some Windows base-image foreign layers) are *not* copied: `if storageCommon.IsNonDistributable(blob.MediaType) { continue }` (`pkg/extensions/sync/destination.go:204-207`). Such images don't round-trip through Zot intact.

The net effect: an image served *out of* default-mode Zot has the **same config blob and the same layer blobs** (identical digests) but a **different manifest digest and index digest** from the origin — purely because the manifest/index bodies are re-encoded with OCI media-type strings. Skip-if-already-synced compares against the *upstream* digest (`CanSkipImage`, `destination.go:56-83`), so this is stable across re-syncs — but it is stably *different from the origin*. (An image whose origin manifest is *already* OCI — as most current Docker Hub library images now are — has nothing to convert and passes through with its digest intact; the rewrite only bites images still served as Docker schema2.)

**The byte-preserving alternative — `PreserveDigest`.** Zot can sync without converting: the sync registry option `PreserveDigest: true` ("sync without converting", `pkg/extensions/config/sync/config.go:37`). With it set, Zot stores and serves the original schema2 bytes, so digests match the origin exactly. The catch: it is only legal when schema2 compatibility is enabled — startup *fails* otherwise: `if regCfg.PreserveDigest && !config.IsCompatEnabled()` → `"can not use PreserveDigest option without enabling http.Compat"` (`pkg/cli/server/root.go:1578-1584`). Compat is turned on with `http.compat: ["docker2s2"]` (`pkg/api/config/config.go`, value `DockerManifestV2SchemaV2`). On-demand sync enters at `pkg/extensions/sync/on_demand.go`.

**Implications for this lab.** This rewriting is exactly why the committed design ([§7.4](02-caching-design.md#74-the-committed-design-nginx-cache-zot-as-verification-oracle)) keeps Zot **off** the serving path. A client doing an *unmodified* pull by digest (`image@sha256:<origin-digest>`) — which the project treats as a hard requirement — would get a **404 from default-mode Zot**, because Zot only knows the image under its *recomputed* digest. nginx's byte-exact pass-through ([§13.2](#132-nginx-wildcard-oci-catch-all)) has no such problem. The only Zot-in-path variant compatible with the unmodified-pull rule is `PreserveDigest` + `docker2s2` compat. Whether that trade is worth making — letting Zot ingest/normalize and caching its output downstream in nginx — is weighed in [§7.5](02-caching-design.md#75-revisit-would-zot-in-path-be-better) and tested for real in the experiment described there.

### 13.2 nginx wildcard OCI catch-all

The **primary shared cache**: one nginx vhost on `:8085` serving any registry via the `ns=` param, with the **Option-1 CDN-redirect-following** pattern so it actually caches blob bytes (not just forwards the 307). The load-bearing detail is **blob keying purely on the digest** ([§7.2](02-caching-design.md#72-what-nginx-only-must-replicate-by-hand)) so the same blob is stored **once** across all repos and namespaces:

```nginx
proxy_cache_path /var/lib/cache/nginx/default
                 levels=1:2 keys_zone=cache_default:100m
                 max_size=100g inactive=30d use_temp_path=off;

# Pull the digest out of /blobs/sha256:<hex> so blobs dedup cross-repo.
map $uri $blob_digest { ~/blobs/(?<d>sha256:[0-9a-f]+)$  $d; default ""; }

server {
    listen 8085 ssl;                                       # TLS to clients (§11.5)
    ssl_certificate     /etc/nginx/cache/cache-server.crt; # shared cache cert, cache CA;
    ssl_certificate_key /etc/nginx/cache/cache-server.key; # same on every cache vhost (constants.cacheTls)
    resolver 1.1.1.1 ipv6=off valid=300s;
    location = /health { return 200 "ok\n"; }

    # Blobs: immutable, content-addressed → key ONLY on the digest.
    location ~ ^/v2/.+/blobs/sha256: {
        set $ns $arg_ns;
        proxy_pass https://$ns;
        proxy_ssl_server_name on;
        proxy_cache cache_default;
        proxy_cache_key "blob:$blob_digest";          # NOT $ns/$uri → cross-repo dedup
        proxy_cache_valid 200 206 30d;
        proxy_cache_lock on;
        proxy_buffering off;
        proxy_intercept_errors on;
        recursive_error_pages on;
        error_page 301 302 303 307 308 = @follow_cdn;
        add_header X-Cache-Status $upstream_cache_status;
        add_header X-Cache-Upstream-Time $upstream_response_time;   # origin fetch on a MISS
    }

    # Manifests + everything else: per-name, key on ns:uri.
    location / {
        set $ns $arg_ns;
        if ($ns = "") { return 404; }              # no ns= → let containerd fall through
        proxy_pass https://$ns;
        proxy_ssl_server_name on;
        proxy_cache cache_default;
        proxy_cache_key "$ns:$request_method:$uri";   # NB: $args excluded
        proxy_cache_valid 200 30d;
        proxy_cache_lock on;
        proxy_intercept_errors on;
        recursive_error_pages on;
        error_page 301 302 303 307 308 = @follow_cdn;
        add_header X-Cache-Status $upstream_cache_status;
        add_header X-Cache-Upstream-Time $upstream_response_time;   # origin fetch on a MISS
    }
    location @follow_cdn {
        set $cdn_url $upstream_http_location;
        proxy_pass $cdn_url;
        proxy_cache cache_default;
        proxy_cache_key "blob:$blob_digest";          # digest only; signed CDN args dropped
        proxy_cache_valid 200 206 30d;
        proxy_cache_lock on;
    }
}
```

Two load-bearing details: (1) `proxy_cache_key` **excludes `$args`** — the signed CDN query string differs every pull, so including it would defeat caching; and (2) blobs key on **`blob:sha256:<digest>` only** — not `$ns:$uri` — so an identical blob pulled under different repos or namespaces collapses to a single cache entry (matches the client hash key in [§11.2](04-client.md#112-consistent-hash-router-and-cache-keys) and the dedup requirement in [§7.2](02-caching-design.md#72-what-nginx-only-must-replicate-by-hand)). Manifests stay keyed by `$ns:uri` because they are per-name and by-tag is mutable.

### 13.3 nginx apt cache

One nginx vhost on `:8086` caching Ubuntu's HTTP apt repos. apt packages are individually GPG-signed, so the cache can't tamper undetected — plain HTTP proxy caching is safe and simple:

```nginx
proxy_cache_path /var/lib/cache/nginx/apt
                 levels=1:2 keys_zone=cache_apt:50m
                 max_size=50g inactive=30d use_temp_path=off;

server {
    listen 8086 ssl;                                       # TLS to clients (§11.5; shared cache cert)
    ssl_certificate     /etc/nginx/cache/cache-server.crt;
    ssl_certificate_key /etc/nginx/cache/cache-server.key;
    resolver 1.1.1.1 ipv6=off valid=300s;
    location = /health { return 200 "ok\n"; }

    location / {
        # apt sends absolute-form requests to a proxy: GET http://archive.ubuntu.com/...
        # nginx forwards by Host: header; we cache .deb immutably, indices briefly.
        proxy_pass http://$http_host;
        proxy_cache cache_apt;
        proxy_cache_key "$http_host$request_uri";
        proxy_cache_valid 200 206 30d;       # .deb are content-addressed by version → immutable
        location ~ (InRelease|Release|Packages|Sources)(\.(gz|xz|bz2))?$ {
            proxy_pass http://$http_host;
            proxy_cache cache_apt;
            proxy_cache_key "$http_host$request_uri";
            proxy_cache_valid 200 5m;        # indices change → short TTL
        }
        add_header X-Cache-Status $upstream_cache_status;
        add_header X-Cache-Upstream-Time $upstream_response_time;   # origin fetch on a MISS
    }
}
```

`stub_status` is enabled on a localhost-only vhost so `nginx-prometheus-exporter` ([§19](07-tuning-observability.md#19-observability-prometheus)) can scrape it.

### 13.4 Docker Hub CDN handling

Docker Hub serves blob bytes from CloudFront/Cloudflare/R2 via short-lived **307 signed-URL redirects** (see `design.md` §6 for the full story). Handling in the focused design:

- **Shared nginx** (`:8085`) uses the Option-1 `@follow_cdn` pattern above to follow the 307 itself and cache the body **keyed by digest**, so a warm digest costs zero upstream traffic regardless of signature drift.
- **The client nginx hash hop is unaffected** — the 307 happens *inside* the shared nginx, never on the client→cache hop, so consistent hashing on the digest keeps working and the survivor refills cleanly on failover.
- **Zot** (the oracle) follows the 307 server-side via its `containers-image` HTTP client and stores by digest — used by the differential test ([§20](08-operations.md#20-build-and-run-workflow)) to confirm nginx's redirect-follow matches.
